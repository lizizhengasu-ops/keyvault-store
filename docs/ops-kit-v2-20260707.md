# Ops-Kit 最终方案 (修订版 v2)

> 研究来源: GitHub 开源项目 (wp-maintenance-automation, InfraMonitorNginx, db_backups, aws-s3-backup, Website-Monitoring,
>   Linux hardening scripts) + WordPress 官方 WP-CLI 文档 + 行业标准实践
> 日期: 2026-07-07
> 定位: 独立站持续运维工具箱 (Bash 脚本 + cron)

---

## 一、架构

```
每台服务器独立部署 ops-kit
/
├── /opt/ops-kit/
│   ├── config.env          # 配置文件 (域名/路径/告警邮箱/S3密钥)
│   ├── scripts/
│   │   ├── backup.sh       # 备份 (每日 03:00)
│   │   ├── backup-restore.sh  # 恢复
│   │   ├── monitor.sh      # 监控 (每 5 分钟)
│   │   ├── security-scan.sh # 安全扫描 (每日 06:00)
│   │   └── update.sh       # 自动更新 (每周日 04:00)
│   ├── logs/               # 运行日志
│   └── install.sh          # 一键安装
└── /var/backups/            # 备份存储 (本地)
```

## 二、模块

### 2.1 backup.sh

| 项目 | 内容 |
|------|------|
| 范围 | MySQL 全库 + wp-content/uploads/ + themes + plugins + wp-config.php |
| 存储 | 本地 (/var/backups/) + 远程 S3 (可选) |
| 加密 | 支持 GPG 加密备份包 |
| 保留 | 每日保留 7 天 / 每周保留 4 周 / 每月保留 12 个月 |
| 验证 | 备份后自动校验 md5 + 可配置的自动恢复测试 |
| 恢复 | backup-restore.sh 一键恢复 |
| 告警 | 失败 → 邮件通知 |
| 参考 | db_backups (★5), aws-s3-mysql-auto-backup (★3), wp-maintenance-automation |

### 2.2 monitor.sh

| 项目 | 内容 |
|------|------|
| 站点存活 | curl HTTP 200 检测, 失败 → 告警 |
| SSL 证书 | openssl 检查到期日 < 14 天 → 告警 |
| 磁盘空间 | df 检查磁盘使用率 > 80% → 告警 |
| PHP 错误 | 检查 PHP error log 是否有新错误 |
| 响应时间 | curl -w %{time_total} 记录页面加载时间 |
| MySQL 状态 | mysqladmin ping 检测数据库连接 |
| 内存/CPU | 负载 > 4.0 时告警 (可选) |
| 频率 | 每 5 分钟 (状态检测) / 每小时 (错误日志) |
| 告警 | 失败 → 邮件通知 (复用 marketing-kit 的 SendGrid) |
| 状态输出 | /var/log/ops-kit/status.json (供 dashboard 读取) |
| 参考 | InfraMonitorNginx (★4), Website-Monitoring-using-Bash-Script (★1) |

### 2.3 security-scan.sh

| 项目 | 内容 |
|------|------|
| 文件权限 | 修复: 目录 755, 文件 644, wp-config 600 |
| 核心校验 | wp core verify-checksums, 检查 WordPress 核心文件完整性 |
| 可疑文件 | 扫描 wp-content/ 中最近 7 天新增的可疑文件 (.php.txt, .gif.php 等) |
| 用户审计 | 检查管理员用户列表, 发现未知管理员 → 告警 |
| XML-RPC | 检查 wp-config 中是否禁用 XML-RPC |
| 文件编辑 | 检查 DISALLOW_FILE_EDIT 是否配置 |
| 登录保护 | 检查 wp-login.php 是否被爆破 (分析 access log) |
| 数据库前缀 | 检查 wp_ 前缀是否被修改 (安全建议: 使用非默认前缀) |
| 频率 | 每日 06:00 |
| 参考 | WPScan (★5700+), Linux 安全硬化脚本, WP-CLI 官方文档 |

### 2.4 update.sh

| 项目 | 内容 |
|------|------|
| 预检查 | 更新前自动运行 backup.sh 创建全量备份 |
| 预览 | wp plugin update --all --dry-run 列出待更新项 |
| 核心更新 | wp core update |
| 插件更新 | wp plugin update --all --minor (只更新小版本, 降低风险) |
| 主题更新 | wp theme update --all |
| 后续检查 | wp core verify-checksums + curl 200 检测站点是否正常 |
| 回滚 | 如果检查失败 → 自动从备份恢复 |
| 报告 | 生成更新报告 (成功/失败/回滚) |
| 频率 | 每周日 04:00 |
| 参考 | wp-maintenance-automation (2026-07-06), WP-CLI 官方文档 |

## 三、配置 (config.env)

```bash
# 站点配置
SITE_URL="https://aliali.local"
SITE_NAME="KeyStarter"

# 备份配置
BACKUP_DIR="/var/backups"
BACKUP_RETENTION_DAILY=7
BACKUP_RETENTION_WEEKLY=4
BACKUP_RETENTION_MONTHLY=12
BACKUP_ENCRYPT=true
BACKUP_ENCRYPT_KEY="your-gpg-key"

# S3 远程存储 (可选)
S3_ENABLED=false
S3_BUCKET="s3://my-backups"
S3_ACCESS_KEY=""
S3_SECRET_KEY=""

# 告警配置
ALERT_EMAIL="admin@keystarter.com"
ALERT_WEBHOOK="https://marketing-kit:3333/webhook/deploy-event"

# 站点健康
MONITOR_INTERVAL=300
SSL_WARN_DAYS=14
LOAD_WARN_THRESHOLD=4.0
DISK_WARN_PERCENT=80
```

## 四、告警

| 级别 | 触发条件 | 通知 |
|------|---------|------|
| CRITICAL | 站点宕机 / 备份失败 / 更新回滚 | 邮件 + 短信(webhook) |
| WARNING | SSL < 14天 / 磁盘 > 80% / 可疑文件 | 邮件 |

告警通道复用 marketing-kit 的 Resend/SendGrid webhook 接口。

## 六、安装

```bash
# 服务器上一键安装
curl -fsSL https://keystarter.com/ops-kit/install.sh | bash

# 或手动
git clone https://github.com/keystarter/ops-kit.git /opt/ops-kit
cd /opt/ops-kit
cp config.env.example config.env
# 编辑 config.env
crontab -e (模板在 install.sh 中)
```

## 九、与最初方案的区别 (修订内容)

| 原方案 | 修订后 | 原因 |
|--------|-------|------|
| 仅 MySQL + wp-content 备份 | 增加 wp-config、加密、校验、md5 验证 | 行业标准 + GitHub 参考 |
| 无备份验证 | 增加自动恢复测试 (可选) | 防止"有备份不能用" |
| HTTP 200 检测 | 增加响应时间、PHP 错误、MySQL 状态 | 更全面的健康监控 |
| 基础安全扫描 | 增加用户审计、XML-RPC 检查、登录保护 | WordPress 特有安全风险 |
| 更新无回滚 | 增加后检查 + 自动回滚 | 防止更新导致站点宕机 |
